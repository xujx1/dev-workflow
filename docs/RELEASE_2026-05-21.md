# dev-workflow v3.3.0 发布公告

这次发布是一版反馈驱动的 Harness 稳定性升级。

感谢 @liumingz 提供《dev-workflow 使用分享.pdf》。这份反馈不是简单的使用体验记录，而是把 dev-workflow 放到真实研发流程里拆解后提出的系统性问题：Orchestrator 边界、状态事实源、知识库保鲜、项目级扩展、代码风格画像、mock-first 例外、OpenSpec 触发、自动修复停止条件、项目级安装和模型路由。本次 `v3.3.0` 主要围绕这些问题做了收敛。

## 本次重点

### 1. Doctor 诊断入口

新增 `/dev-workflow:doctor` 和 `.workflow/scripts/doctor*.js`，用于在流程启动前或升级后检查：

- 配置文件是否完整。
- `execution-state.md` 是否合法。
- 关键产物是否存在。
- Beads、插件版本、知识库新鲜度是否异常。
- 飞书权限与 scope 是否满足当前阶段。

目标是把“流程跑到一半才暴露问题”前移到启动前诊断。

### 2. 状态一致性与安全微操作

新增 Orchestrator 安全微操作白名单、审计日志、执行状态修复、产物校验和 reconcile 差异报告。

这让 Orchestrator 可以执行低风险、可机械化、可审计的基础设施动作，例如创建缺失目录、补齐状态字段、校验产物存在性和生成差异报告，同时继续禁止它直接生成业务产物、实现代码、测试代码和 Review 结论。

### 3. 复杂度分级与风险升级

`02-implementation-plan` 和 `03-code-gen-tdd` 增加 nano / lite / full 分级思路，并支持运行中按风险升级：

- 涉及资金、库存、状态机、数据一致性、权限边界时提高流程严谨度。
- 多应用、跨团队、外部 API 或消息契约变更时触发更强治理步骤。
- 高风险链路可追加 OpenSpec、GitNexus 或更强模型。

### 4. OpenSpec 多维触发

OpenSpec 不再只依赖人日阈值，而是扩展为规模、接口变更、跨系统依赖和业务域风险的组合判断。

这使小人日但高风险的接口变更也能进入规格化流程，降低上下游契约漂移的概率。

### 5. 项目级扩展与风格画像

新增组织级、项目集级、项目级配置模板与配置合并器，支持：

- 配置继承与覆盖。
- locked 字段保护。
- 配置来源追踪。
- 项目风格画像、业务画像、Review 画像。
- 多技术栈模板集：Java Spring Boot、Node.js Express、Python FastAPI。

这部分主要回应“如何让 AI 写出像本项目的代码”和“业务系统特性应该放在哪一层”的问题。

### 6. Harness 自身回归测试

新增 `.workflow/tests`：

- L1 契约测试：检查 Skill / Agent / 规则结构和基本约束。
- L2 流程模拟：覆盖缺 PRD、自动修复达到上限、模型路由缺失、reconcile warn 等异常路径。
- L3 E2E 预留：用于重大版本前验证完整链路。

以后 dev-workflow 自身升级不再只靠人工阅读 diff，而有了可运行的回归测试入口。

### 7. 模型路由与自动修复停止条件

新增模型路由解析器，支持按阶段选择 quick / deep / writing 模型类别，并在连续失败、Review 阻塞、高风险链路等场景自动升级。

同时补充自动修复停止条件，避免在同一错误上无限循环，或把修复范围扩散到非本需求文件。

## 升级建议

升级后建议运行：

```bash
cd .workflow/tests
npm install
npm run test:all
```

如只想快速验证模型路由，可在仓库根目录运行：

```bash
node acceptance-test.js
```

完整变更见 `docs/CHANGELOG.md`，反馈原文和讨论材料已归档在 `usage-feedback /0521`。
