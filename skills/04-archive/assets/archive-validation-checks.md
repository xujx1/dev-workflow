# 归档验证检查与质量标准

## 质量标准

- 归档报告是正式产物，禁止"草稿/草案"后缀
- 归档知识库为"最终实现归档"，冲突时以 `archive_code_ref` 代码事实为准
- 若最终实现与方案不一致，必须补充"最终实现差异"摘要
- 报告禁止保留 `{score}`、`{prd_coverage}`、`TODO` 等未替换占位符
- 报告必须包含 6 节结构 + 核心指标汇总表

---

## 自动检查（子代理执行）

- [ ] 若本次需求存在 OpenSpec 产物，则 `openspec/changes/archive/` 下已包含本次变更目录；若不存在 OpenSpec，则已显式跳过 Step 4-1
- [ ] `ai_commit_hash` 已完成 `git rev-parse --verify --quiet` 校验；若不可解析，归档报告节4已明确写 `—` 与原因说明，未误写 `0%`
- [ ] 归档前已锁定当前分支最新代码快照 `archive_code_ref`
- [ ] 知识库已按本次变更类型至少更新 1 类目标文档（例如 `03_核心流程与逻辑层.md`、`02_架构与设计层.md`）
- [ ] 若更新了 `03_核心流程与逻辑层.md`，其内容为摘要式增量更新，而非大段实现细节堆叠
- [ ] `{feature_dir}/archive-report.md` 存在且包含 6 节内容和核心指标汇总表
- [ ] 归档报告头部已记录 `ai_commit_hash` 与 `archive_code_ref`；若两者不同，已说明归档以当前分支最新代码快照为准
- [ ] 报告已上传飞书，`feishu_report_url` 非空
- [ ] 归档 `git commit` 已执行，`archive_commit_hash` 非空
- [ ] 报告顶部核心指标表中已填充 `MRD / PRD / 技术方案 / AI代码采纳率 / 测试覆盖率` 等指标
- [ ] 报告节 6 已包含基于主要读取路径的自然语言总结
- [ ] 报告中不存在 `{score}`、`{prd_coverage}`、`{tech_coverage}`、`{code_adoption}`、`{line_coverage}`、`TODO` 等未替换占位符

## 需人工确认（无确认门，归档后直接输出汇总）

- 归档报告中的核心指标是否符合预期
- 知识库更新是否完整反映了本次变更

## 验证未覆盖

- 飞书 Wiki 目录结构（需人工检查目录位置是否正确）
- 知识库内容的业务准确性（需业务方 review）
