## 1. `.workflow` 是干嘛的？

`.workflow` 是 dev-workflow 的 **Harness 运行时与治理目录**。它不是业务代码目录，而是把“AI 研发流程本身”工程化、可检查、可恢复、可测试的基础设施目录。

可以按六类理解：

| 模块 | 目录 / 文件 | 作用 |
| --- | --- | --- |
| 运行时脚本 | `.workflow/scripts/` | Doctor 诊断、配置合并、状态修复、reconcile、模型路由、PRD/技术方案校验、知识库保鲜检测 |
| 安全边界 | `.workflow/orchestrator-safe-ops.yml` | 定义 Orchestrator 能做哪些低风险微操作，哪些产物绝对不能由 Orchestrator 直接生成 |
| 配置解析 | `.workflow/resolved-config.json` | 保存 org / workspace / project 三层配置合并后的运行时结果，并保留来源信息 |
| 项目画像 | `.workflow/profiles/` | 保存业务画像、代码风格画像、Review 画像，让实现、测试、Review 更贴近具体项目 |
| 模板集 | `.workflow/templates/` | 按技术栈提供代码生成、测试生成、Review 清单和 artifact contract 模板 |
| 回归测试 | `.workflow/tests/` | Harness 自身的 L1 契约测试、L2 流程模拟、L3 E2E 预留，防止升级把流程改坏 |

一句话：`.workflow` 是 **dev-workflow 自己的工程系统**，用于支撑流程预检、恢复、配置继承、风险治理、测试验证和版本升级。

## 2. 本次升级主要改动点

### 2.1 诊断与恢复能力

- 新增 `/dev-workflow:doctor` 入口和 `agents/doctor/doctor-agent.md`。
- 新增 `.workflow/scripts/doctor.js`、`.workflow/scripts/doctor-check.js`，检查配置、执行状态、关键产物、Beads、飞书权限、插件版本、知识库新鲜度和模型路由。
- 新增 `.workflow/scripts/reconcile.js` / `reconcile.py`，生成 `execution-state`、Beads、git diff 的差异报告。
- 新增 `.workflow/scripts/fix-execution-state.js`、`validate-artifacts.js`，支持低风险状态修复和产物存在性校验。

### 2.2 Orchestrator 安全微操作

- 新增 `.workflow/orchestrator-safe-ops.yml`。
- 明确允许：创建缺失目录、修复 `execution-state.md` 的基础字段、校验产物存在性、生成 reconcile 报告。
- 明确禁止：生成 PRD、技术方案、业务代码、测试代码、OpenSpec、Review 结论、知识库修改等正式产物。
- 新增 `.workflow/ops-audit.log` 作为审计日志入口。

### 2.3 复杂度分级与风险升级

- `skills/02-implementation-plan/SKILL.md` 增加 nano / lite / full 三档。
- 技术方案阶段根据规模、风险、协作范围写入 `execution-state.md` 的 `complexity` 字段。
- 风险升高时允许从低档自动升级到高档；降级必须人工确认。
- `agents/tech-design/assets/tech-design-template.md` 同步加入复杂度和风险升级判断。

### 2.4 OpenSpec 触发条件精细化

- `plugins/openspec/SKILL.md` 从“人日阈值”扩展为多维 OR 规则。
- 新增触发维度：新增 API、修改 API 入参/出参、跨系统接口依赖、业务域强制配置。
- 支持 `openspec_required: always / never / default`。
- `java-impl-agent` 在存在 `tasks.md` 时优先消费 OpenSpec 任务。

### 2.5 mock-first 例外协议

- `skills/03-code-gen-tdd/SKILL.md` 增加 mock-first 例外协议。
- 明确哪些场景可申请真实依赖或 Testcontainers：复杂 SQL、事务/锁、第三方 SDK 状态机、消息幂等、文件系统等。
- 要求例外必须在技术方案“测试策略”中声明。
- 单元测试仍默认保持 mock-first，例外走集成测试层。

### 2.6 自动修复停止条件

- `skills/03-code-gen-tdd/assets/decision-trees.md` 增加 `STOP_01` ~ `STOP_05`。
- 覆盖最大尝试次数、无进展、引入新失败、编译连续失败、超时等停止条件。
- 新增 `.workflow/autofix-history.md` 记录修复尝试历史。

### 2.7 项目级扩展与画像

- 新增 `skills/00-init/assets/org-config-template.json` 和 `workspace-config-template.json`。
- 新增 `.workflow/scripts/config-resolver.js`，支持 org / workspace / project 三层配置合并。
- 新增 `.workflow/profiles/business-profile.md`、`style-profile.md`、`review-profile.md`。
- `agents/java-impl`、`agents/java-review`、`agents/tech-design`、`agents/testcode-gen` 增加对项目画像、业务特性和技术栈配置的读取。

### 2.8 多技术栈模板

- 新增 `.workflow/templates/java-spring-boot-v2/`。
- 新增 `.workflow/templates/nodejs-express-v1/`。
- 新增 `.workflow/templates/python-fastapi-v1/`。
- `examples/00-init/mrd-to-code-config.json` 和配置模板支持 `tech_stack` / `template_set` 路由。

### 2.9 模型路由

- 新增 `.workflow/scripts/model-resolver.js`。
- 支持按阶段映射模型类别：`quick`、`deep`、`writing`、`default`。
- 支持单模型 baseline 降级。
- 支持连续失败、Review 阻塞等风险升级到更强模型。
- 新增 `acceptance-test.js` 验收模型路由。

### 2.10 Harness 自身回归测试

- 新增 `.workflow/tests/`。
- L1：Skill 契约测试。
- L2：异常流程模拟测试。
- L3：E2E 预留。
- 新增 `pre-upgrade-hook.sh`，用于版本升级前阻断破坏性变更。

## 4. 还不是“完全闭环”的部分

这次升级已经把 0521 反馈中的主要问题都转成了可执行能力或流程规则，但以下部分仍建议在后续真实项目中继续验证：

1. **知识库保鲜质量**：已有 `freshness-check.js` 和 `freshness.yml`，但保鲜分数是否能真实反映业务知识过期，还需要更多项目样本校准。
2. **三层安装升级命令**：配置继承和解析已落地，`mrd-upgrade` 仍偏脚手架，后续可继续完善“只升级 org / workspace / project”的真实命令体验。
3. **画像生成质量**：已有 profile 文件和读取链路，但风格画像、业务画像是否足够影响实现和 Review，需要在真实代码生成中观察效果。
4. **OpenSpec 触发误判**：多维规则已落地，后续需要根据误触发 / 漏触发案例调整规则权重。

结论：0521 的演进问题本次已经完成 **主体升级**；其中问题 3、问题 9 属于“基础能力已落地，仍需真实项目继续校准”的半闭环项。
