# 和作者讨论的问题

本文档用于逐条讨论《dev-workflow 使用分享.pdf》第 4 部分“我希望和作者讨论的问题”。

讨论方式建议：

1. 先确认问题是否成立。
2. 再讨论可选方案。
3. 最后沉淀一个阶段性结论，必要时转成后续改进项。

## 问题 1：Orchestrator 的边界应该硬到什么程度？

### 背景

当前设计要求 Orchestrator 只调度、不产出。这能防止主会话越权，也让产物来源更清晰。

但实践中会遇到一些轻量操作，例如：

- 补齐缺失目录。
- 修复 `execution-state.md` 中的元数据。
- 做产物存在性校验。
- 做状态 reconcile。

如果这些操作全部绕 Agent，流程会变重。

另一个关键约束是主 Agent 的上下文预算。很多模型上下文只有 200k 左右，如果把目录修复、状态修复、产物校验、reconcile 等流程全部写进 Skill，Skill 本身会持续膨胀。主 Agent 在执行长流程时还要同时携带用户需求、阶段状态、错误信息、产物路径和规则约束，很容易被流程性说明挤占上下文。

### 可讨论方向

是否引入“安全微操作白名单”：

- 允许 Orchestrator 修复 `execution-state.md` 的元数据。
- 允许创建缺失目录。
- 允许做产物存在性校验和状态 reconcile。
- 继续禁止生成业务产物、代码、测试和 Review 结论。

### 待讨论问题

- 哪些操作可以被定义为“安全微操作”？
- 安全微操作是否需要记录审计日志？
- 白名单是写死在 Skill 中，还是做成可配置项？
- 哪些修复流程应该沉淀为脚本或轻量命令，而不是继续写进 Skill 上下文？
- 如何控制 Skill 的长度，避免主 Agent 在长流程中被流程说明打满上下文？

### 阶段性结论

Orchestrator 的边界应该硬在“不产出业务判断和工程产物”，而不是硬到连低风险基础设施动作都不能做。

Skill 应该定义边界、原则、入口和验收标准，而不是把所有修复流程都塞进去。可机械执行、低风险、可审计的修复动作应下沉为脚本、命令或 Orchestrator 的安全微操作白名单，例如创建目录、修复状态元数据、校验产物存在性、生成 reconcile 报告。这样既能减少 Agent 调度成本，也能避免 Skill 过长导致主 Agent 上下文被打满。

## 问题 2：`execution-state`、Beads、文件产物谁是事实源？

### 背景

实践中可能出现三者不一致：

- `execution-state` 写了 phase done，但关键产物缺失。
- Beads 标记 done，但本地文件没有生成。
- 文件存在，但 `execution-state` 没有推进。

当前 dev-workflow 已经有产物校验的规则要求和 Phase 检查清单，但主要还是流程约束和 LLM 执行契约；还缺少一个统一的 artifact validator / reconcile 机制，能够自动对齐 `execution-state`、关键产物、Beads 状态和 git diff / commit hash。

### 可讨论方向

是否需要统一的 reconcile 机制，在每次 resume 前自动比对：

```text
读取 execution-state
检查关键产物
检查 Beads 状态
检查 git diff / commit hash
生成差异报告
自动修复低风险不一致
高风险不一致交给用户确认
```

### 待讨论问题

- 什么属于低风险不一致，什么必须人工确认？
- reconcile 应该是独立命令，还是每次 resume 自动执行？
- `execution-state` 优先时，产物校验失败后应该回退 phase，还是阻断并等待人工确认？

### 阶段性结论

倾向于 `execution-state` 优先，但必须校验产物。

原因是 `execution-state` 是长流程恢复的入口，它负责告诉 Orchestrator 当前流程声称走到哪里、下一步应该做什么、关键产物在哪里。如果不以它为优先入口，resume 时就需要重新扫描大量文件和任务状态，流程恢复会变慢，也更容易消耗主 Agent 上下文。

但 `execution-state` 不能单独证明阶段完成。它更像流程游标和索引，不是最终验收证据。阶段是否真的完成，必须用关键产物、Beads 状态和必要时的 git diff / commit hash 做校验。

可以形成这样的规则：

```text
execution-state 决定从哪里恢复
关键产物决定阶段是否真的完成
Beads 决定任务协作状态是否一致
git diff / commit hash 决定代码事实是否可追溯
```

因此，每次 resume 前应该先读取 `execution-state`，再按 state 中声明的 phase 和产物路径做 reconcile。低风险不一致可以自动修复，例如补充缺失的状态字段、同步产物路径、把未推进的 state 推进到已存在且校验通过的产物阶段。高风险不一致应该阻断流程并请求人工确认，例如 state 标记完成但核心产物缺失、产物结构不满足契约、Beads 已关闭但代码 diff 不存在。

## 问题 3：知识库应该“简洁”还是“丰富”？

### 背景

知识库太简，会导致 Agent 缺少业务语义；知识库太大，又会造成 token 浪费和注意力污染。

当前项目已经采用 L0 / L1 / L2 三层结构，方向上是合理的。真正需要讨论的不是“要不要分层”，而是默认注入多少、按需读取多少，以及如何控制长流程中的 token 消耗。

现在也很难预先定义一套通用标准，判断什么知识库最适合真实业务系统。不同业务系统的高频链路、历史坑点、接口复杂度、测试风险都不一样，知识库质量更适合通过实践结果持续反馈，而不是一开始就写成固定模板。

### 可讨论方向

可以引入“简洁丰富”的评估指标：

- `CONTEXT.md` 是否覆盖核心实体、接口、流程。
- `api-index` 是否覆盖主要入口。
- 核心流程文档是否覆盖高频需求链路。
- 最近失败的 Agent 是否能反向暴露知识库缺口。
- 是否需要知识库覆盖率或知识库保鲜度指标。

其中，“知识库覆盖率”主要用于衡量知识库是否足够丰富，例如核心实体、接口入口、高频业务流程、关键状态流转、高风险历史链路是否已经被知识库覆盖。

“知识库保鲜度”主要用于防止知识库过时，避免知识库一次性生成后长期不更新。它可以关注最近需求归档是否反哺、代码结构变化后知识库是否同步、PRD / 技术方案 / 实现偏差是否已沉淀。

### 待讨论问题

- L0 / L1 / L2 的边界怎么定义？
- 知识库更新应该按需求归档触发，还是定期扫描触发？
- 如何避免知识库变成不可维护的大文档？
- 如何限制默认注入内容，避免知识库把主 Agent 或子 Agent 上下文打满？
- 如何评估知识库带来的 token 消耗是否值得？
- 如何通过 04 归档、PRD 准确性、技术方案准确性和代码实现偏差反哺知识库？
- 知识库覆盖率应该覆盖哪些维度，才能衡量“丰富”而不是单纯衡量字数？
- 知识库保鲜度应该由归档流程自动更新，还是由定期巡检发现过时内容？

### 阶段性结论

知识库建设应优先控制上下文和 token 成本，而不是追求一次性“足够丰富”。

当前三层结构可以保留：L0 负责短小稳定的默认导航，L1 负责按 Agent 职责路由读取，L2 负责沉淀更完整的业务背景、历史问题和归档资料。关键原则是默认少注入、需要时再读取，避免知识库本身成为长流程上下文膨胀的主要来源。

知识库覆盖率和知识库保鲜度可以作为两个不同指标：覆盖率衡量知识库是否丰富，保鲜度衡量知识库是否持续跟随业务和代码演进。二者不应该和默认上下文大小混为一谈；知识库可以在 L1 / L2 层足够丰富，但默认注入仍然保持克制。

对于“什么知识库最适合业务系统”，不建议一开始制定过细标准。更可行的方式是通过实际交付反哺知识库：

- 04 归档阶段把本次需求的新业务规则、接口变更、测试策略和历史坑点沉淀回知识库。
- PRD 偏差用于反推领域概念、业务角色、流程边界是否缺失。
- 技术方案偏差用于反推架构约束、接口契约、状态流转、数据模型是否缺失。
- 代码实现偏差用于反推项目风格、典型实现模式、异常处理、日志和测试习惯是否缺失。

阶段性方向是：知识库先保持分层和克制，通过需求交付结果持续增量补齐，而不是提前构建一个庞大但不一定会被正确读取的知识库。

## 问题 4：各业务系统的扩展性应该如何设计？

### 背景

原问题 4“项目规范应该放在哪个层级”和问题 5“如何让 AI 写出像本项目的代码”本质上是同一个问题：Harness 如何在保持通用流程的同时，允许不同业务系统扩展自己的规则、风格和上下文。

代码约束和项目规范可能同时存在于组织、项目集和具体项目中。仅靠通用 coding-standard 不够，AI 还需要理解当前业务系统的技术栈、模块边界、惯用写法和历史坑点。

内部推广时也已经收到类似反馈：不同业务系统不只需要通用 Java 开发手册，还会有自己的业务特性和 Review 关注点，例如数据一致性保证、资金链路、状态流转、幂等、补偿、权限边界等。如果当前 Review 只加载通用 Java 规范，就很难覆盖这些业务系统特有风险。

目前内部也有一些部门基于当前 plugin 体系做了自己的增强插件，例如 hook 或其他扩展能力。这说明扩展诉求是真实存在的，但现在还缺少统一标准来定义：业务特性应该写在哪里、什么时候加载、由哪个 Agent 使用、如何验证它确实生效。

### 可讨论方向

可以把扩展性拆成三层：

| 层级 | 内容 |
| --- | --- |
| 组织级 | 通用安全、日志、异常、代码风格、测试底线 |
| 项目集级 | 技术栈、框架版本、通用中间件、缓存和 MQ 规范 |
| 项目级 | 具体业务领域、模块边界、历史坑点、高风险链路 |

是否支持规则继承：

- 组织规则。
- 项目集规则。
- 项目规则。
- 按优先级合并，冲突时项目级覆盖项目集级。

项目级扩展还应包括项目风格画像，例如 `style-profile.md`，由知识库阶段自动生成或持续维护：

- 常见分层写法。
- 日志风格。
- 异常处理风格。
- DTO / Entity / BO 转换习惯。
- 测试命名和断言风格。
- 注释写法和边界。

当前初步想法是把业务特性维护在知识库中，并在代码实现、Code Review 等阶段按需加载。例如：

- 实现阶段加载业务系统的领域规则、接口约束、数据一致性要求。
- Review 阶段加载业务系统的高风险检查点和业务特性清单。
- 测试阶段加载业务系统的关键场景、历史缺陷和一致性校验策略。
- hook / plugin 扩展用于在特定时机做自动检查、提示或阻断。

### 待讨论问题

- 规则冲突如何检测？
- 规则继承是否需要锁版本？
- Agent 执行时如何知道当前适用的是哪一组规则？
- 风格画像应该自动生成还是人工维护？
- `java-impl` 是否应该优先搜索同类代码，再按风格画像实现？
- `java-review` 是否应该检查实现是否偏离业务系统惯用写法？
- 业务系统特性应该沉淀为知识库、规则、hook/plugin，还是三者组合？
- 代码实现和 Review 阶段应该按什么标准决定加载哪些业务特性？
- 如何验证业务特性加载后确实影响了实现、Review 和测试，而不是只停留在文档里？

### 阶段性结论

问题 4 和问题 5 可以合并为“各业务系统的扩展性问题”。

Harness 的通用流程应该保持稳定，但规则、知识和风格必须允许业务系统扩展。建议采用“组织级默认规则 + 项目集级技术规则 + 项目级业务和风格扩展”的结构，并支持继承、覆盖和版本锁定。

其中，问题 5 提到的“写出像本项目的代码”不应作为独立问题处理，而应作为项目级扩展能力的一部分。项目级扩展不仅包括业务知识和历史坑点，也应该包括风格画像、典型实现模式、测试习惯和 Review 检查点。

当前可以先把业务系统特性放在知识库中维护，并在代码实现、Review、测试阶段按需加载；同时允许业务部门通过 hook/plugin 做更强约束。但这件事暂时不宜贸然全面推进，因为还缺统一标准：业务特性文档的格式、路由规则、加载时机、优先级、冲突处理、效果验证都需要先约定清楚。

因此，下一步更适合先定义一个最小标准，例如 `business-profile.md` / `style-profile.md` / `review-profile.md` 三类项目级扩展文件，再选择一两个业务系统试点，验证这些扩展是否真的能提升 PRD、技术方案、代码实现和 Review 的准确性。

## 问题 6：mock-first 是否应该有例外协议？

### 背景

mock-first 能提升测试速度和稳定性，也能避免测试连接真实 DB / Redis。

当前系统的职责更偏向单元测试，单元测试主要解决开发提效问题。为了控制时间成本和 token 成本，当前才设定 mock-first 规则：优先通过快速、稳定、低依赖的单元测试验证本次代码改动。

同时，内部也在同步开发集成测试自动化插件。集成测试更适合覆盖 Spring 容器、AOP、配置绑定、动态代理、序列化链路、真实中间件交互和端到端数据一致性等问题。因此，这里讨论的重点不是让当前系统直接承担所有集成测试能力，而是明确当前单元测试闭环与集成测试插件之间的边界和衔接。

### 可讨论方向

定义例外协议：

- 什么条件下允许 Spring context 测试。
- 什么条件下允许 Testcontainers。
- 单元测试和集成测试覆盖率是否分开统计。
- 例外是否需要在 `tech-design` 或 `test_spec` 中显式说明。
- 当前系统发现 mock-first 无法覆盖的风险时，是否只输出集成测试建议，而不是直接执行集成测试。
- 集成测试自动化插件应如何读取 `tech-design`、`test_spec` 或业务 profile 中声明的风险点。

### 待讨论问题

- mock-first 的默认边界是什么？
- 哪些场景由当前单元测试闭环覆盖，哪些场景转交集成测试插件？
- 当前系统是否需要生成 integration-test-handoff，而不是内联执行集成测试？
- 集成测试失败后的修复闭环由哪个系统负责？

### 阶段性结论

当前系统仍应坚持 mock-first，定位是单元测试提效，而不是完整替代集成测试体系。

mock-first 的核心原因不是认为集成测试不重要，而是为了控制开发提效场景下的时间、稳定性和 token 成本。单元测试闭环应该优先验证本次代码改动的业务分支、异常分支、边界条件和局部逻辑，不默认拉起 Spring context、真实 DB、真实 Redis 或 Testcontainers。

集成测试能力可以由独立自动化插件承接。当前系统需要做的是在 `tech-design`、`test_spec` 或业务 profile 中识别并标记“mock-first 无法证明”的风险点，例如容器装配、事务边界、SQL 映射、MQ 幂等、缓存一致性、序列化兼容、数据一致性保证等，然后生成清晰的集成测试交接信息。

阶段性方向是：当前系统负责单元测试和风险识别；集成测试插件负责高成本、高依赖、跨组件验证。两者通过明确的风险标记和 handoff 文件衔接，而不是把集成测试逻辑直接塞进当前 mock-first 闭环。

## 问题 7：OpenSpec 触发条件是否应该从“人日”扩展为“风险”？

### 背景

按需求估算人日触发 OpenSpec 很实用，但规模不一定等于风险。

只按人日触发会漏掉一类低人日但高风险的需求，例如改一个状态枚举、接口字段、MQ 消息体、权限判断或幂等逻辑。它们实现成本可能不高，但失败代价很大。

### 可讨论方向

OpenSpec 触发可以从“规模阈值”扩展为“规模 + 风险阈值”。

风险维度包括：

- 跨应用接口变更。
- 涉及资金、权限、状态机、幂等、补偿。
- 涉及对外 API 或消息契约。
- 涉及多人并行开发。
- 涉及高风险历史链路。

可以进一步拆成三类触发：

```text
OpenSpec 触发 = 规模触发 + 风险触发 + 协作触发
```

- 规模触发：需求估算超过某个人日阈值，例如 ≥5 人日。
- 风险触发：涉及资金、权限、状态机、幂等、补偿、数据一致性、对外 API、MQ 契约、跨应用接口。
- 协作触发：多人并行、跨团队、跨应用、需要前后端或上下游对齐。

触发后也不一定都走完整 OpenSpec，可以分级：

| 档位 | 适用场景 |
| --- | --- |
| none | 小文案、小 if 分支、小校验，不触发 OpenSpec |
| lite | 单应用普通业务逻辑，或命中少量风险点，只生成轻量规格 |
| full | 跨应用、多人协作、高风险历史链路、命中多个高风险点 |

### 待讨论问题

- 风险评分由谁给出？
- 风险阈值是否可以配置？
- 低人日但高风险需求是否强制走 OpenSpec？
- 风险触发后默认走 lite，还是高风险直接走 full？
- OpenSpec 触发规则是否应该进入 `.mrd-to-code-config.json`，由项目配置阈值和风险关键词？

### 阶段性结论

OpenSpec 不应该只按人日触发。人日衡量规模，风险衡量失败代价，协作衡量沟通成本。低人日但高风险的需求，也应该触发 OpenSpec 或至少触发轻量规格化。

阶段性方案是采用“规模 + 风险 + 协作”的触发机制，并支持 none / lite / full 分级，避免 OpenSpec 成为所有需求的固定成本。

默认策略可以是：风险触发后先走 lite；命中多个高风险点、跨应用、多人并行或历史高风险链路时升级为 full。这样既能覆盖低人日高风险需求，也能控制流程成本。

后续可以把触发规则配置化，例如在 `.mrd-to-code-config.json` 中维护：

```json
{
  "openspec": {
    "enabled": true,
    "person_day_threshold": 5,
    "risk_triggers": [
      "permission",
      "state_machine",
      "idempotency",
      "mq_contract",
      "external_api",
      "data_consistency"
    ],
    "risk_mode": "lite_first"
  }
}
```

## 问题 8：自动修复什么时候必须停？

### 背景

重试次数上限是必要的，但停止条件还可以更语义化。

自动修复的风险在于：如果只按次数控制，AI 可能在错误方向上连续尝试，或者为了让测试通过不断扩大改动范围，最后把非本需求的问题也吞进来。停止条件应该同时考虑失败类型、改动范围、收益变化和产物契约冲突。

### 可讨论方向

可以定义自动修复停止条件：

- 连续两次修复同一段代码但失败原因不变，停止。
- 测试失败从业务断言变成环境异常，停止。
- 修复开始扩大到非本需求文件，停止。
- 覆盖率提升收益很低，停止。
- `test_spec`、技术方案、实现三者出现冲突，停止并请求人工判断。

### 待讨论问题

- 停止条件应该由 Runner 判断，还是 Orchestrator 判断？
- 停止后应该生成什么诊断报告？
- 是否允许用户手动批准继续修复？
- 停止条件应该只写在 Phase 5 Runner 中，还是作为全流程通用机制？
- 自动修复停止后，是否应该把失败分类写回 `execution-state`，并生成下一步人工处理建议？

### 阶段性结论

自动修复必须同时有“次数上限”和“语义化停止条件”。

次数上限解决成本和无限循环问题，但不能判断修复方向是否已经跑偏。语义化停止条件用于识别：失败原因没有变化、失败类型发生漂移、改动范围扩散、收益变低、或者 `test_spec`、技术方案、实现之间出现冲突。

建议形成几类停止信号：

- 重复失败：连续两次修改同一段代码，但失败原因、失败用例或错误栈没有变化。
- 失败漂移：原本是业务断言失败，修复后变成环境异常、编译异常或依赖问题。
- 范围扩散：修复开始修改非本需求文件、公共基础设施代码或未在变更清单中的模块。
- 收益不足：覆盖率或通过率提升很低，但改动复杂度持续增加。
- 契约冲突：`test_spec`、技术方案、实现、Review 结论之间出现明显不一致。
- 事实不足：需要业务判断、环境配置、外部依赖或历史上下文才能继续判断。

停止后不应该静默失败，也不应该继续自动试错。Runner 或 Orchestrator 应该把失败分类、已尝试动作、当前证据、建议下一步写回 `execution-state`，并生成人工确认点。用户可以手动批准继续，但继续前应明确允许的改动范围和下一轮目标。

## 问题 9：Harness 安装应该在用户级还是项目集级？

### 背景

核心 Harness 如果散落在个人环境里，团队协作和 CI 复现都会受影响。

如果 Harness 的规则、Skill、Agent、插件版本都放在个人环境里，同一个需求在不同机器上可能跑出不同流程，CI / 自动化环境也难以复现。问题排查时也很难区分是模型问题、流程问题，还是个人环境差异。

### 可讨论方向

倾向于：

- 核心 Harness 项目集级管理。
- 用户级只保留运行时能力。
- 项目集级做版本管理和变更审计。
- 用户级用于试用、升级和个人定制。

可以按三层拆分：

```text
用户级：runtime / auth / local tools / personal preferences
项目集级：workflow / skills / agents / rules / plugin policy / version lock
项目级：business-profile / style-profile / review-profile / app knowledge base
```

同时保留自下而上的演进机制：如果某个业务系统在项目实践中沉淀出通用的 plugin、skill、hook 或 profile 标准，并且在多个项目中验证有效，就可以上收集成到项目集级或核心 Harness 中。

内部已经有插件版本自动更新机制，所以这里不是要从零设计插件升级能力，而是要讨论自动更新机制如何与项目集级锁版本、灰度发布、兼容性校验和回滚策略配合。

当前自动更新机制可以基于本地 `VERSION` 与 Git 上的 `VERSION` 做版本比对。如果本地版本落后太多，系统可以自动拉取新版本并执行热部署更新，降低团队成员长期停留在旧插件版本上的概率。

### 待讨论问题

- 项目集级 Harness 如何锁版本？
- 用户级能力和项目级规则如何合并？
- 插件市场和项目配置之间如何协作？
- 业务系统自定义 plugin / skill 在什么条件下可以上收为通用能力？
- 上收后的通用能力如何做版本管理、兼容性验证和回滚？
- 现有插件版本自动更新机制如何支持项目集级锁版本、灰度和回滚？
- 本地 `VERSION` 与 Git `VERSION` 的落后阈值如何定义，超过阈值后是否强制热部署？

### 阶段性结论

Harness 的“协议部分”应该项目集级管理，Harness 的“运行时部分”可以用户级安装。

项目集级负责流程一致性、规则版本、团队共享和 CI 复现；用户级负责本地执行能力、授权、调试和个人增强；项目级负责业务系统自己的知识库、业务 profile、风格画像和 Review 扩展。

阶段性方向可以概括为：核心 Harness 项目集级锁版本，业务系统项目级扩展，用户级只装执行器。

同时，项目级扩展不是孤岛。如果在推广过程中发现某些业务系统自定义的 plugin、skill、hook 或 profile 标准具备通用价值，并且在多个业务系统中验证有效，就应该逐步上收，集成进项目集级或核心 Harness，形成可复用能力。

考虑到内部已经有插件版本自动更新机制，后续重点不是“如何让插件更新”，而是“如何让更新可控”。核心 Harness 可以使用项目集级版本锁保证团队一致性，同时通过自动更新机制承接通用 plugin / skill 的升级、灰度、兼容性验证和必要时回滚。

具体机制上，可以通过本地 `VERSION` 与 Git 上的 `VERSION` 比对判断版本差距。当本地版本落后超过设定阈值时，自动触发插件更新和热部署；当项目集级锁版本要求稳定时，则应优先遵守锁版本，并把可用更新提示给用户或维护者。

## 问题 10：什么时候该用什么模型？

### 背景

模型选择如果每次靠人判断，流程会不稳定，也不利于团队复用。

模型路由多模型编排方案：动态感知当前用户环境中的可用模型，按任务 Category 选择模型，通过子 Agent 调用参数注入模型，并提供降级链和 baseline 兜底。

但需要考虑一个现实约束：很多使用方只有单模型，或者没有完整执行初始化流程。因此模型路由不能假设所有人都有多 Provider、多模型可选，也不能因为缺少模型配置就中断主流程。

### 可讨论方向

可以把模型选择纳入 workflow 配置：

| 阶段 | 模型倾向 |
| --- | --- |
| 知识库初扫 | 成本优先，轻量模型 |
| MRD 澄清 / PRD | 文档结构化强的模型 |
| 技术方案 | 推理更强的模型 |
| 核心代码实现 | 编码能力强的模型 |
| Review / 风险分析 | 推理和审查能力强的模型 |
| 归档总结 | 轻量模型 |

也可以按风险自动升级：

- 跨域需求。
- 高风险链路。
- 连续失败。
- 多次 Review BLOCK。

可以采用 Category 路由，而不是直接在流程里写死具体模型：

| Category | 说明 | 适用场景 |
| --- | --- | --- |
| deep | 深度推理 | 技术方案、复杂实现、测试规格设计 |
| quick | 快速执行 | Review 规则检查、测试代码生成、归档整理 |
| writing | 文档写作 | PRD、文档润色、归档报告 |
| default | 默认兜底 | 通用任务、未配置场景 |

模型解析优先级可以参考：

```text
用户显式覆盖
→ Agent 专属配置
→ Agent → Category 映射
→ Category 首选模型
→ Category 降级链
→ 项目默认模型
→ 环境 baseline 模型
→ 全员可用兜底模型
```

对于只有单模型的使用方，模型路由应退化为 baseline 单模型模式：所有 Agent 使用同一个可用模型，流程不中断，只在高价值阶段给出轻量提醒，提示用户可以配置多模型路由以优化效果和成本。

### 待讨论问题

- 模型路由应该按阶段配置，还是按任务风险动态选择？
- 模型升级是否需要用户确认？
- 如何记录不同模型的成本和效果？
- 单模型使用方是否默认使用 baseline，并跳过 Category 优化？
- 可用模型列表应该从本地配置读取，还是通过 `/v1/models` 动态感知并缓存？
- 未配置模型路由时，是静默兜底还是在关键 Stage 友好提醒？

### 阶段性结论

模型路由应该配置化、Category 化，并支持风险驱动升级；但必须兼容单模型使用方。

阶段性方案：从用户本地模型环境读取 baseline，动态获取可用模型列表，按 Agent → Category 映射选择模型，通过子 Agent 调用参数注入模型。Category 负责表达任务能力需求，例如 deep、quick、writing、default；具体模型由配置和可用性决定。

对多模型使用方，按 Category 选择更合适的模型，并在跨域、高风险、连续失败、多次 Review BLOCK 等场景自动升级。对单模型使用方，直接退化为 baseline 单模型模式，所有 Agent 使用同一个可用模型，流程不中断。

模型路由不应该强绑定具体模型名称，而应该优先表达能力等级和降级策略。即使用户没有执行初始化，或 `.mrd-to-code-config.json` 中缺少模型配置，也应该读取环境 baseline 或使用全员可用兜底模型继续执行，并只在关键阶段给出轻量提醒。

后续还应该记录模型使用效果，例如阶段、Category、实际模型、token 消耗、耗时、是否一次通过、是否触发重试、Review BLOCK 数和人工修改量，用数据反向优化模型路由策略。

## 我建议的后续演进方向

这一部分不是要立即实现代码，而是先把演进方向讨论清楚。整体主线可以概括为：

> 从流程能跑，演进到流程可诊断、可分级、可回归、可扩展。

### 方向 1：增加 doctor 能力

#### 目标

解决“环境、配置、状态和产物是否健康”的问题，减少流程跑到一半才暴露配置缺失、权限不足、状态不一致或插件版本落后的情况。

#### 可检查内容

- 配置文件是否完整，例如 `.mrd-to-code-config.json`。
- Agent / Skill / Rules 路径是否可解析。
- 飞书 MCP 权限、身份、scope 是否满足当前流程。
- `execution-state`、关键产物、Beads 状态是否一致。
- OpenSpec、GitNexus、Beads、模型路由等插件是否可用。
- 本地 `VERSION` 与 Git 上的 `VERSION` 是否差距过大，是否需要自动更新或热部署。
- 知识库是否过期，是否长期未被 04 归档反哺。

#### 讨论重点

- doctor 是独立命令，还是每个 Stage 开始前做轻量检查？
- 哪些问题可以自动修复，哪些必须阻断并提示用户？
- doctor 的输出应该是简单通过 / 失败，还是分级报告？
- 如何做到低用户感知：正常通过时静默，只在异常、风险或需要用户决策时提示？

#### 阶段性判断

doctor 优先级较高。它可以承接前面讨论过的事实源 reconcile、产物校验、插件版本、模型配置和飞书权限检查，是后续流程稳定性的基础。

更倾向于每个 Stage 开始前做轻量检查，而不是要求用户主动执行一个重 doctor 命令。正常情况下检查静默通过，减少用户感知；只有发现配置缺失、权限不足、状态不一致、插件版本落后过多或需要人工决策时，才输出简短提示或阻断。

独立 doctor 命令仍然可以保留，用于排障、初始化后自检、版本升级后验证或用户主动诊断。但日常主流程应该内置轻量 pre-stage doctor，避免把健康检查变成额外操作负担。

更合适的设计是把 doctor 定义为一种 capability，而不是单一 command 或单一 Agent：

```text
doctor capability
├── pre-stage lightweight checks：主流程内置，默认静默，异常时提示或阻断
├── doctor-agent：负责复杂诊断、聚合证据和生成报告
└── doctor command：手动排障入口，用于初始化后自检、升级后验证和问题反馈
```

其中，pre-stage doctor 是日常主路径，负责低成本检查并返回结构化结果，例如 `pass`、`warn`、`block`、`autofix_done`。doctor-agent 只在需要复杂诊断时调用，避免把大量诊断逻辑塞进 Orchestrator 上下文。doctor command 可以保留，但不作为日常推广的主入口，避免增加用户操作负担。

第一阶段可以先落地轻量版本：每个 Stage 前做少量关键检查，正常静默，异常时给出简短提示；复杂报告和完整 command 可以后续逐步增强。

### 方向 2：增加需求复杂度分级

#### 目标

解决“是不是所有需求都要跑完整流程”的问题，降低小需求使用成本，同时保证高风险需求有足够治理。

#### 建议分级

| 档位 | 适用场景 | 流程倾向 |
| --- | --- | --- |
| nano | 文案、字段名、简单校验、小 if 分支 | tech-only + 单元测试 |
| lite | 常规单应用需求 | 知识库 + 技术方案 + TDD |
| full | 跨应用、高风险、多人协作需求 | OpenSpec + GitNexus + 完整归档 |

#### 讨论重点

- 分级是否只由人日决定，还是由“规模 + 风险 + 协作”共同决定？
- nano / lite / full 是否应该进入 `.mrd-to-code-config.json` 配置？
- 分级错误时如何升级或降级流程？
- 技术方案阶段是否应该输出推荐复杂度档位和依据，并写入配置或 `execution-state`？

#### 阶段性判断

复杂度分级应该和问题 7 的 OpenSpec 触发逻辑保持一致：人日衡量规模，风险衡量失败代价，协作衡量沟通成本。默认不应让所有需求都承担 full 流程成本。

这个分级适合在技术方案阶段落地。`tech-design` 可以根据需求规模、风险点、协作范围输出推荐档位，例如 `complexity_level=nano|lite|full`，并说明判断依据。后续 `03-code-gen-tdd`、OpenSpec、GitNexus、归档等流程根据该档位决定启用哪些能力。

可考虑在技术方案或状态文件中写入类似配置：

```yaml
complexity:
  level: lite
  reasons:
    - single_app_change
    - no_external_contract_change
  risk_triggers:
    - data_consistency
  collaboration_scope: single_team
  recommended_flow:
    openspec: lite
    gitnexus: optional
    archive: standard
```

如果后续实现或 Review 发现风险高于技术方案判断，流程应允许从 nano 升级到 lite，或从 lite 升级到 full；但降级应更谨慎，避免低估需求风险。

### 方向 3：增加 Harness 自身回归测试和自身开发能力

#### 目标

解决“Harness 自己改动后会不会把流程改坏”的问题。当前 Harness 已经不是简单 prompt，而是一套工程系统，因此也需要自己的回归测试。

同时，也可以研究 Harness 如何支持“自身开发”：用同一套需求、方案、实现、Review、测试、归档机制来迭代 Harness 自己。也就是说，Harness 不只服务业务系统研发，也应该能反过来服务自身插件、Skill、Agent、Rules 的开发和验证。

#### 可测试内容

按三层建设：

- **L1 静态契约测试**：检查 Skill / Agent / Rules 是否满足输入、阶段、产物、状态更新、权限边界等基本契约。涵盖 Skill contract test（输入/产出/状态）和 Agent fixture test（固定输入、关键字段验证）。
- **L2 流程模拟测试**：用伪造的 `execution-state`、产物缺失、Beads 状态不一致、模型配置缺失等场景，验证 resume、stop、warn、reconcile 等行为是否符合预期。
- **L3 端到端测试**：用固定 example 需求验证 golden path（完整跑通）、failure path（缺文件/权限失败/覆盖率不足时正确停下）和 resume path（上下文中断后能否正确恢复）。

#### 讨论重点

- Harness 回归测试应该跑到什么粒度，是只测契约还是端到端试跑？
- 是否需要固定一组 example 需求作为 golden cases？
- 回归测试失败是否阻止插件版本自动更新或热部署？
- Harness 自身开发是否也应该走 nano / lite / full 分级？
- Skill、Agent、Rules 的变更是否需要自己的 test_spec、review-profile 和归档报告？
- 如何避免 Harness 自测消耗过多 token 和时间，反而拖慢自身迭代？

#### 阶段性判断

Harness 自身回归测试可以作为自动更新和热部署的安全网。尤其是在插件版本自动更新机制已经存在的情况下，需要用回归测试保证“更新可控”。

第一阶段不建议直接追求完整端到端自动化，而应优先建设低成本的静态契约测试和流程模拟测试，先保证 Skill、Agent、State、产物、Beads、插件版本和停止条件这些流程协议不被改坏。端到端 golden path / failure path 可以作为后续增强。

Harness 自身开发能力也值得继续研究。长期看，Harness 应该能用自己的流程开发自己：需求变更进入技术方案，Skill / Agent / Rules 修改后触发契约测试和流程模拟测试，验证通过后再进入版本更新和热部署。这会让 Harness 从“人工维护的一组提示词和插件”演进成“可自我迭代的工程系统”。

### 方向 4：增加项目风格画像和业务扩展 profile

#### 目标

解决“不同业务系统如何长得像自己”的问题。它承接问题 4 / 5：业务系统需要扩展自己的规则、风格、Review 检查点和业务特性。

#### 建议 profile

- `business-profile.md`：业务特性、数据一致性、状态机、幂等、权限边界、高风险链路。
- `style-profile.md`：代码风格、分层习惯、日志、异常、DTO 转换、测试命名。
- `review-profile.md`：业务系统特有 Review 检查点，例如数据一致性、资金链路、状态流转、补偿逻辑。

其中，项目风格画像不只是格式问题，而是帮助 AI 写出“像本业务系统”的代码。它应该覆盖：

- 分层习惯：Controller / Service / Domain / Repository 如何拆分职责。
- 命名习惯：DTO、BO、VO、Entity、Assembler、Converter 的命名方式。
- 日志习惯：哪些链路必须打日志，是否必须包含 bizId / orderNo / userId 等业务标识，哪些敏感字段禁止打印。
- 异常习惯：业务异常、外部依赖异常、参数校验异常如何处理。
- 转换习惯：对象转换使用手写 converter、Assembler、MapStruct 还是 builder。
- 测试习惯：mock 风格、断言风格、测试命名、given-when-then 组织方式。
- 注释习惯：普通代码不写解释性注释，但状态机、资金、库存、数据一致性逻辑必须说明业务原因。

`style-profile.md` 可以由知识库阶段从现有代码中抽样生成初版，再通过 04 归档持续修正。当 Review 发现“不像本项目”、人工修改较多，或者某类实现反复偏离项目习惯时，应把这些反馈沉淀回 `style-profile.md`。

#### 讨论重点

- profile 应该由知识库阶段自动生成，还是由业务系统人工维护？
- `java-impl`、`java-review`、`testcode-gen` 分别应该加载哪些 profile？
- 如何验证 profile 真的影响了实现、Review 和测试，而不是只停留在文档里？
- 业务系统沉淀出的通用 profile、plugin、skill、hook，何时上收进项目集级或核心 Harness？

#### 阶段性判断

profile 标准适合先试点，不宜一开始全面铺开。可以先选一两个业务系统，把业务特性、代码风格和 Review 规则沉淀成最小 profile，再观察 PRD、技术方案、代码实现和 Review 准确性是否提升。

项目风格画像应该作为项目级扩展 profile 的一部分，由知识库阶段生成初版，再通过 04 归档持续修正。它不是为了统一格式，而是为了让 AI 在实现、测试和 Review 时优先遵循当前业务系统的惯用写法和高风险约束。

### 演进优先级建议

建议顺序：

1. 先做 doctor，因为它降低所有流程的失败成本。
2. 再做复杂度分级，因为它降低小需求使用成本。
3. 同步试点 profile 标准，因为这是内部推广中的核心反馈。
4. 最后补 Harness 自身回归测试，把前面能力固化成可持续演进的保障。

## 讨论总结

10 个问题均已完成讨论，每个问题都有阶段性结论。

### 当前开放的行动项

各问题结论中仍有若干尚未落地的待讨论点，优先级较高的包括：

1. **reconcile 机制**（问题 2）：统一的 artifact validator / reconcile 机制还未实现，是流程稳定性的重要基础。
2. **业务扩展 profile 标准**（问题 4）：`business-profile.md` / `style-profile.md` / `review-profile.md` 还缺少格式约定，需要先选一两个业务系统试点。
3. **OpenSpec 触发规则配置化**（问题 7）：风险关键词和阈值可以进入 `.mrd-to-code-config.json`，但需要确认字段格式和评估主体。
4. **Harness 安装层级**（问题 9）：项目集级锁版本、灰度和回滚机制还需要结合现有自动更新机制具体设计。

### 后续演进方向

已在"我建议的后续演进方向"部分讨论完成，建议执行顺序：

1. **doctor 能力**：降低所有流程的失败成本，优先级最高。
2. **复杂度分级**：降低小需求使用成本。
3. **profile 试点**：对接内部推广中的核心反馈。
4. **Harness 自身回归测试**：固化演进保障，在插件自动更新机制之上加安全网。
